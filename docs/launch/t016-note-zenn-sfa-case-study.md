# Lark Base × LARC でSFAを自作した話 — AI権限設計で詰まった実録

> Note / Zenn 掲載用 | 著者: 林駿甫（合同会社みやび）

---

## はじめに

「Salesforceは高い。でもスプレッドシートで営業管理は限界。」

そう思って、Lark Base × AI エージェントでSFA（営業支援システム）を自作しました。

結果的に**30件のタスクをキューに積み、23件を自動処理**できるようになりました。権限絡みのインシデントはゼロ。

この記事では、構築で一番詰まった**「AIに何をさせるか」「何をさせないか」の権限設計**を中心に、実装の流れを書きます。

---

## なぜ Lark Base を選んだか

| 比較軸 | Salesforce | Notion | **Lark Base** |
|-------|-----------|--------|--------------|
| 初期コスト | 高（月数万〜） | 中 | 低（プラン内） |
| API 操作 | 複雑 | 制限あり | **lark-cli で完結** |
| IM 統合 | なし | なし | **IM/承認/Wiki と同一テナント** |
| AI エージェント連携 | 要カスタム | 要カスタム | **LARC で統合設計済み** |

Lark は IM・Base・Drive・Approval が同一テナントで動く。AI エージェントが「メッセージ送信 → Base更新 → 承認フロー起動」を一気通貫で処理できる唯一のプラットフォームでした。

## システム構成

### Lark Base 5テーブル構成

```
contacts    — 顧客（リード〜顧客）
companies   — 企業
deals       — 案件（パイプライン管理）
activities  — 活動ログ（電話・DM・ミーティング）
lead_scores — AIスコアリング結果
```

### LARC 統合レイヤー

```
[AI エージェント]
      ↓
LARC ingress queue（enqueue → in_progress → done/blocked）
      ↓
lib/ingress.sh — 権限チェック + ゲート判定
      ↓
lark-cli — Base / IM / Approval API
      ↓
Lark（Base更新 / IM送信 / 承認フロー起動）
```

LARC は「AIと Lark の間の通関」です。何が通れて何が通れないかを、ここで一元管理します。

## 権限設計で詰まった3つのポイント

### 1. Bot Token と User OAuth の境界

Lark APIには「Bot でできること」と「ユーザー本人しかできないこと」が明確に分かれています。

```
✅ Bot でできる
  - IM メッセージ送信（as bot）
  - Lark Base レコードの読み書き
  - Wiki ページ読み取り

❌ User OAuth が必要（Bot 不可）
  - 承認タスクの承認・却下
  - 承認申請の取り下げ
  - 承認担当者の変更
```

最初、承認フローの「承認アクション」までAIにやらせようとして、403エラーで詰まりました。

Lark はこれを**意図的に**制限しています。承認は「本人の意思確認」が前提なので、Botに代理させられない。

LARC ではこれを `blocked` 状態として設計しました。

```bash
# 承認申請の作成まではAIが実行
larc approve create --task submit_expense --amount 50000

# → Lark Approval フローが起動
# → キューは blocked に遷移
# → 人間が Lark UI で承認/却下
# → larc ingress resume <id> でキューを再開
```

「AIができないことをAIがやろうとして黙って失敗する」より、「できないことを明示して人間に委ねる」設計の方が、運用して1ヶ月でその価値が分かりました。

---

### 2. 最小権限を事前に知る方法

SFA構築の初期、「このタスクに何のスコープが要るか」を調べるたびに Lark のドキュメントを読んでいました。

LARC の `larc auth suggest` を使うと、自然言語から必要スコープを推論できます。

```bash
$ larc auth suggest "顧客にフォローアップのIMを送り、活動ログをBaseに記録する"

タスクタイプ: send_crm_followup
必要スコープ:
  - im:message:send_as_bot    (identity: bot)
  - bitable:app               (identity: bot / user)
ゲート: preview — 送信前に内容を確認してください
推奨プロファイル: backoffice_agent
```

「送信前にpreviewゲートが挿入される」のが重要です。AIが自動生成したメッセージを、人間が確認してから送る。このワンクッションで、誤送信を防げます。

---

### 3. 自動化してよいタスクとダメなタスクの分類

SFAの業務を全部洗い出して、4象限に分類しました。

```
              リスク低          リスク高
              
自動OK   ┌──────────────┬──────────────┐
         │  即実行       │  preview確認  │
         │  ・活動ログ記録│  ・IM送信     │
         │  ・スコア更新 │  ・Base一括更新│
         ├──────────────┼──────────────┤
要承認   │  通知後実行   │  承認必須     │
         │  ・オンボード │  ・契約書送付 │
         │    グループ作成│  ・案件クローズ│
         └──────────────┴──────────────┘
```

この分類を `config/gate-policy.json` に落とし込んで、コードに判定ロジックを書かない設計にしました。ポリシーを変えたいときはJSONを編集するだけ。

## 実装：オンボーディング自動化

新規リードが `contacts` テーブルに登録されると、5分毎のdaemon がピックアップして自動処理します。

```
contacts.stage = "01.グループ作成依頼中"
  ↓ daemon (5分毎)
lark-cli im +chat-create --name "みやび × {名前}"
  ↓
stage → "02.グループ作成済" + チャットID保存
  ↓
ウェルカムメッセージ送信（Bot）
  ↓
stage → "03.ウェルカム送信済" + 開始日時記録
```

**ゲート設定**: グループ作成は `none`（自動）、ウェルカムメッセージは `preview`（内容確認後送信）。

最初は両方自動にしていましたが、テスト中にテスト用リードに本番メッセージを送ってしまったので、ウェルカムメッセージはpreviewゲートを挟むように変更しました。

## 実装：コンテキストリフレッシュ

30分毎に動くスクリプトが、CRMの最新状態を AI のワーキングメモリに同期します。

```bash
# context-refresh.sh (30分毎 cron)
1. contacts / deals / activities を Base API で取得
2. IM グループの最新メッセージを取得
3. LARC queue の状態取得
4. ~/.larc/cache/workspace/main/memory/YYYY-MM-DD.md に追記
5. larc memory push → Lark Base に永続化
```

これで AI が「今日どの案件が動いているか」「どのリードが未返信か」をセッション横断で把握できます。

## 3ヶ月の運用結果

| メトリクス | 値 |
|-----------|-----|
| キュー総タスク数 | 30件 |
| 完了（done） | 23件 |
| ブロック（承認待ち） | 4件 |
| 失敗（fail） | 3件 |
| 権限インシデント | **0件** |
| 手動介入が必要だった操作 | 承認タスクのみ（設計通り） |

失敗した3件はすべて「Lark API のレート制限」が原因で、権限設計とは無関係でした。`larc ingress retry` で再実行して解決。

## やってよかったこと・失敗したこと

### ✅ やってよかった

- **preview ゲートを惜しまない**: IM送信・Base一括更新には必ずpreviewを挟む。最初は面倒に見えるが、誤操作を1件防ぐだけで元が取れる
- **`blocked` を怖がらない**: 承認待ちで止まるのは設計通り。「AIが詰まった」ではなく「人間の判断待ち」として扱う
- **ポリシーをJSONで管理する**: gate-policy.json を見れば「どの操作がどの承認レベルか」が一覧できる。コードに散らばらない

### ❌ 失敗したこと

- **最初から全自動にしようとした**: オンボーディングメッセージを全自動にして誤送信。preview ゲートの価値を身をもって学んだ
- **Bot/User OAuth の境界を後から学んだ**: 承認APIで403を食らってから調べた。最初に `larc auth suggest` で確認すべきだった
- **キューのリカバリ設計を後回しにした**: in_progress のまま詰まったタスクへの対処（`larc ingress recover`）を後から追加した。最初から設計しておけばよかった

## まとめ

Lark Base × LARC でSFAを作るのに、コーディング経験は必要でしたが、Salesforceのような専門知識は不要でした。

一番時間を使ったのは「AIに何をやらせるか」の設計です。ここを最初に丁寧にやると、運用が安定します。

**権限設計のチェックリスト（実経験から）**:

- [ ] Bot でできること / User OAuth が必要なことを分類した
- [ ] 各タスクに gate（none/preview/approval）を割り当てた
- [ ] `blocked` 状態になったときの人間の対処フローを決めた
- [ ] キューが詰まったときのリカバリ手順を用意した
- [ ] ポリシーをコードではなく設定ファイルで管理している
